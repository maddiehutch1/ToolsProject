# Architecture Document
## Mental Health Companion — Agentic Chat Application

**Version:** 1.0  
**Date:** March 9, 2026  
**Status:** Draft

---

## 1. System Overview

The application is a single-server Python backend (FastAPI) that serves both the static frontend and the `/chat` API. When a user sends a message, the server checks for crisis signals, then runs a LangGraph agent that may invoke one or more tools before streaming the final response back to the browser via Server-Sent Events.

```
Browser (HTML/CSS/JS)
    │
    │  POST /chat  ──►  SSE stream (tokens + tool events)
    │
FastAPI Server
    ├── Crisis detection (pre-LLM gate)
    ├── LangGraph StateGraph (agent loop)
    │       ├── call_model node  ──►  OpenAI gpt-4o-mini
    │       └── tools node
    │               ├── rag_search    ──►  FAISS vector index
    │               ├── web_search    ──►  Tavily API
    │               └── calculator    ──►  simpleeval
    └── Static file serving (index.html, style.css, app.js)

FAISS Index (local disk — two flat files: .faiss + .pkl)
    └── Embedded chunks from knowledge_base/*.md
```

## 3. Request Lifecycle

A complete turn from user input to final rendered response:

```
1.  User types message and submits (Enter or Send button)
2.  app.js  →  POST /chat  { message, session_id }
3.  main.py  →  detect_crisis(message)
        ├── [crisis detected]  →  stream hardcoded crisis event  →  done
        └── [no crisis]  →  continue
4.  main.py  →  append HumanMessage to session history
5.  main.py  →  StreamingResponse(event_generator())
6.  agent.py  →  graph.astream_events(state, version="v2")
7.  LangGraph: call_model node
        →  llm_with_tools.invoke(messages)
        →  OpenAI API returns AIMessage
            ├── [has tool_calls]  →  tools node
            └── [no tool_calls]  →  END  →  stream tokens
8.  LangGraph: tools node
        →  ToolNode dispatches to matching @tool function
        →  Tool executes; returns string result
        →  ToolMessage appended to state
        →  loop back to call_model
9.  call_model sees tool result in context
        →  generates final answer
        →  streams tokens
10. agent.py  →  yields { type: "done" }
11. main.py  →  appends AIMessage to session history
12. app.js   →  renders final bubble; hides spinner
```

## 4. LangGraph Agent Design

### 4.1 State Shape

```python
class AgentState(TypedDict):
    messages: Annotated[list[BaseMessage], operator.add]
```

`messages` accumulates across the entire agent loop. `operator.add` means each node appends to the list rather than replacing it — this is LangGraph's standard reducer for message history.

### 4.2 Graph Topology

```
        ┌─────────┐
  START │         │
   ──►  │  call_  │ ──── tool_calls present? ──► [ tools ] ──┐
        │  model  │                                           │
        │         │ ◄─────────────────────────────────────────┘
        └────┬────┘
             │ no tool_calls
             ▼
           [ END ]
```

| Node | Type | What it does |
|------|------|-------------|
| `call_model` | Custom | Invokes `llm_with_tools.invoke(state["messages"])`. Returns `{"messages": [ai_message]}`. |
| `tools` | `ToolNode` (prebuilt) | Reads `tool_calls` from the last `AIMessage`, dispatches each to the matching `@tool` function, appends `ToolMessage` results to state. |

**Conditional edge** (`should_continue`):
```python
def should_continue(state: AgentState) -> str:
    last_message = state["messages"][-1]
    return "tools" if last_message.tool_calls else END
```

### 4.3 Tool Binding

```python
tools = [rag_search, web_search, calculator]
llm_with_tools = ChatOpenAI(model="gpt-4o-mini", streaming=True).bind_tools(tools)
```

LangChain derives the tool schemas automatically from the `@tool` decorator and function docstrings. No manual JSON schema is required.

### 4.4 System Prompt

The system prompt is prepended as a `SystemMessage` at the start of every `messages` list:

- Establishes role: psychoeducational companion, not a therapist
- Instructs: prefer `rag_search` for any mental health technique or coping strategy question
- Instructs: use `web_search` for current events, recent research, or anything not in the knowledge base
- Instructs: use `calculator` for any numeric scoring (PHQ-9, GAD-7, sleep efficiency)
- Instructs: always cite which tool was used at the end of the response
- Tone guidance: warm, non-judgmental, evidence-based, concise

## 5. RAG Pipeline

### 5.1 Ingest Flow (run once via `backend/ingest.py`)

```
knowledge_base/*.md
        │
        ▼
TextLoader (per file)
        │
        ▼
RecursiveCharacterTextSplitter
  chunk_size=500, chunk_overlap=50
        │
        ▼
List[Document]  (with metadata: { source: filename })
        │
        ▼
OpenAIEmbeddings("text-embedding-3-small")
        │
        ▼
FAISS.from_documents(docs, embedding).save_local("faiss_index")
        │
        ▼
faiss_index/  (two flat files: index.faiss + index.pkl)
```

### 5.2 Retrieval Flow (at runtime, per tool call)

```
User query string
        │
        ▼
vectorstore.as_retriever(search_kwargs={"k": 4})
        │
        ▼
Cosine similarity search over FAISS index
        │
        ▼
List[Document] (top-4 chunks)
        │
        ▼
Format: "[source_filename]\n{page_content}" joined by "---"
        │
        ▼
String returned to LangGraph ToolMessage
```

### 5.3 Knowledge Base Documents

| File | Content Summary |
|------|----------------|
| `cbt_techniques.md` | Core CBT techniques: thought records, cognitive restructuring, behavioral activation, exposure hierarchies |
| `dbt_skills.md` | DBT skill modules: Mindfulness, Distress Tolerance (TIPP, ACCEPTS), Emotion Regulation, Interpersonal Effectiveness (DEAR MAN) |
| `mindfulness_exercises.md` | MBSR-based practices: body scan, breath awareness, mindful observation, STOP technique |
| `grounding_techniques.md` | Somatic grounding: 5-4-3-2-1 senses, box breathing, cold water technique, bilateral tapping |
| `sleep_hygiene.md` | CBT-I guidelines: sleep restriction basics, stimulus control, sleep hygiene rules, sleep efficiency formula |
| `crisis_resources.md` | 988 Suicide & Crisis Lifeline, Crisis Text Line (text HOME to 741741), NAMI, SAMHSA National Helpline |

## 6. SSE Streaming Protocol

All SSE events are emitted as `data: {JSON}\n\n` lines. The frontend parses each `data:` line independently.

### Event Types

| Event Type | Payload Fields | When Emitted |
|-----------|---------------|-------------|
| `token` | `content: string` | Each text chunk from the LLM stream |
| `tool_use` | `tool: string` | When a tool invocation starts (`on_tool_start`) |
| `tool_done` | `tool: string` | When a tool invocation completes (`on_tool_end`) |
| `crisis` | `content: string` | When crisis keywords are detected (pre-LLM) |
| `error` | `message: string` | On any unhandled exception in the stream |
| `done` | _(none)_ | End of the agent turn |

### Example Stream (RAG query)

```
data: {"type": "tool_use", "tool": "rag_search"}

data: {"type": "tool_done", "tool": "rag_search"}

data: {"type": "token", "content": "Here are some grounding techniques "}

data: {"type": "token", "content": "that can help during a panic attack:"}

data: {"type": "done"}
```

### Example Stream (crisis signal)

```
data: {"type": "crisis", "content": "It sounds like you may be in distress. Please reach out immediately: call or text 988 (Suicide & Crisis Lifeline), or text HOME to 741741 (Crisis Text Line). You are not alone."}

data: {"type": "done"}
```

## 7. Crisis Detection

Crisis detection is a **pre-LLM hard gate** — it runs before the agent is invoked. If triggered, the LLM is never called.

### Logic

```python
CRISIS_KEYWORDS = [
    "want to die", "end my life", "kill myself", "suicidal",
    "self-harm", "hurt myself", "not worth living", "can't go on",
    "want to disappear", "no reason to live"
]

def detect_crisis(message: str) -> bool:
    normalized = message.lower()
    return any(kw in normalized for kw in CRISIS_KEYWORDS)
```

### Hardcoded Response

The crisis response is a static string — it does **not** pass through the LLM. It always includes:
- Acknowledgment of distress
- 988 Suicide & Crisis Lifeline (call or text)
- Crisis Text Line (text HOME to 741741)
- NAMI Helpline
- Encouragement to seek professional support

### Frontend Rendering

The `crisis` event type triggers distinct UI treatment:
- Red-bordered card instead of a normal message bubble
- Resource links rendered as clickable anchors
- No tool badge (crisis response is not a tool call)

## 8. Session Management

Session history is maintained **in memory** on the server, keyed by `session_id`.

```python
# backend/main.py
session_store: dict[str, list[BaseMessage]] = {}
```

- `session_id` is a UUID generated by `app.js` on page load
- On each `POST /chat`, the server retrieves or initializes the history for that session
- The `HumanMessage` is appended before the agent runs; the final `AIMessage` is appended after streaming completes
- History is lost when the server restarts (in-scope for MVP; persistence is post-MVP)

## 9. Frontend Architecture

The frontend is a single HTML page with no build step or framework.

### Files

| File | Responsibility |
|------|---------------|
| `index.html` | Page shell; disclaimer banner; message history container; input form |
| `style.css` | Calm color palette; user/agent bubble distinction; tool badge styles; crisis card styles; spinner |
| `app.js` | Session ID generation; form submit handler; SSE stream reader; DOM mutation for streaming; tool badge and crisis card rendering |

### Key Frontend Behaviors

- `sessionId` is generated once via `crypto.randomUUID()` on page load
- Messages are sent via `fetch` (not `EventSource`) so the request body can carry JSON
- The response body is consumed as a `ReadableStream` with a `TextDecoder`
- Each `data:` line is parsed as JSON and dispatched by `event.type`
- The agent message bubble is created before streaming starts; `token` events append text to it in place
- `tool_use` events insert a small badge *above* the in-progress bubble: e.g., `🔍 Knowledge Base`
- A loading spinner is shown between submit and the first `token` event

## 10. Configuration & Environment

### `.env` (never committed)
```
OPENAI_API_KEY=sk-...
TAVILY_API_KEY=tvly-...
PORT=8000
FAISS_INDEX_PATH=faiss_index
```

### `.env.example` (committed)
```
OPENAI_API_KEY=your_openai_api_key_here
TAVILY_API_KEY=your_tavily_api_key_here
PORT=8000
FAISS_INDEX_PATH=faiss_index
```

### Startup Sequence

```
1. python backend/ingest.py       # First time only — builds faiss_index/ (index.faiss + index.pkl)
2. uvicorn backend.main:app --reload --port 8000
3. Open http://localhost:8000
```

The FastAPI server loads the FAISS index from disk at startup. If `faiss_index/` does not exist, the health endpoint reports `"vector_store": "not_loaded"` and the `rag_search` tool returns an error message rather than crashing.
